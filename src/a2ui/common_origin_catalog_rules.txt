**REQUIRED PROPERTIES:** You MUST include ALL required properties for every component, even if they are inside a template or will be bound to data.

- For 'Text', you MUST provide 'text'. Use a plain string or { "path": "/..." } for data binding.
- For 'Alert', you MUST provide 'content'. The 'variant' should be one of: info, success, warning, error.
- For 'EmptyState', you MUST provide 'title'.
- For 'Stack', you MUST provide 'children' as an array of component IDs. The 'direction' should be 'row' or 'column'. The 'gap' should be one of: none, xs, sm, md, lg, xl.
- For 'TextField', you MUST provide 'label'.
- For 'NumberField', you MUST provide 'label'.
- For 'Select', you MUST provide 'label' and 'options'. Each option needs 'value' and 'label'.
- For 'Checkbox', you MUST provide 'label'.
- For 'Button', you MUST provide 'label'. Use 'action' with v0.9 format: {"name":"action_name","context":[{"key":"field","value":"value"}]}. 'onClick' is a legacy fallback.
- For 'Chip', 'FilterChip', 'BooleanChip', you MUST provide 'content'.
- For 'Card', you MUST provide 'title'.
- For 'List', you MUST provide 'children'.
- For 'ListItem', you MUST provide 'primary' (the primary text).
- For 'DateGroup', you MUST provide 'date' and 'children'.
- For 'MoneyDisplay', you MUST provide 'amount' (a number). Defaults: currency=AUD, locale=en-AU.
- For 'TransactionListItem', you MUST provide 'merchant', 'amount', and 'date'. The 'status' should be one of: completed, pending, failed. The 'category' should be one of: shopping, dining, transport, entertainment, bills, other.
- For 'AccountCard', you MUST provide 'accountName' and 'balance'. The 'accountType' should be one of: everyday, savings, credit, mortgage, term-deposit.
- For 'CategoryBadge', you MUST provide 'label'. Do not use 'content'.
- For 'StatusBadge', you MUST provide 'status'. Supported values: pending, completed, failed, cancelled, processing, scheduled.
- For 'TabBar', you MUST provide 'tabs' (array of {id, label}).
- For 'ActionSheet', you MUST provide 'actions' (array of {id, label}).
- For 'Progress', you MUST provide 'value' (0-100).
- For 'Modal', you MUST provide 'title'.
- For 'Skeleton', no extra required props; 'variant' can be rect, circle, or text.

**CHILDREN:** Children are always referenced by ID, never defined inline. Use a 'Stack' component to group multiple children when a component expects a single child.

**DATA BINDING:** Use plain strings/numbers for static values. Use { "path": "/some/key" } to bind to the data model. Paths follow JSON Pointer (RFC 6901) syntax.

**CURRENCY:** All monetary amounts default to AUD. Always use negative numbers for debits/expenses and positive for credits/income.

**ACTIONS:**
- For Button interactions, use: { "name": "actionName", "context": [{ "key": "field", "value": "value" }] }.
- Use { "path": "/field" } inside context values when you need runtime data-model resolution.
- For list-style interactive components (for example ListItem, TransactionListItem), use 'onClick' with the same v0.9 shape: { "name": "actionName", "context": [...] }.
- For form controls (TextField, NumberField, Select, Checkbox), use local onChange with 'eventType' + 'dataPath' to update the local data model.

**PATTERNS:**
Use the following named patterns to structure your UI. Each pattern defines required components, layout rules, and interaction flows.

- account-summary: Requires AccountCard + Stack. Show a MoneyDisplay total. Each AccountCard must have an onClick action.
- transaction-list: When showing results, EVERY TransactionListItem must be wrapped in a DateGroup and have an onClick "view_transaction" action. Show EmptyState when there are no results.
- spending-breakdown: Requires CategoryBadge + MoneyDisplay + Stack. List categories highest-spend first with a MoneyDisplay per category.
- progressive-input: Requires at least one of TextField, NumberField, Select, or Checkbox, plus a primary Button to advance the flow. Always include a cancel/back Button.
- confirmation-flow: Three states — review (summary Card + confirm Button + edit Button), success (Alert + reference number), error (Alert + retry Button).
- search-and-filter: Requires SearchField at the top. Use BooleanChip for filters. Show EmptyState with illustration="search" when no results.
- detail-drilldown: Detail view must have a back Button (action "back_to_list") and wrap content in a Card.
- disambiguation: Show a Text question + two or more Button options when user intent is ambiguous.
- progressive-disclosure: Use when you need one piece of additional context — brief Alert/Text explaining what's needed, 2-3 Button options.
- agentic-consent: Required for destructive actions — warning Alert + confirm Button + cancel Button (cancel must never be styled as 'text' variant).
